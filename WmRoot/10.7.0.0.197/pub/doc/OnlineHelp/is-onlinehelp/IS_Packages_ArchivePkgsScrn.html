Archive Package Screen
Screen Heading: Packages > Management > packagename > Archive
Use the Archive Package screen to create an archive copy of an entire package or a portion of a package. An archive file is a zip file that contains all or some of the files in a package. An archive file is like a distribution file except that it is not made generally available to others. You might create an archive file to make a backup copy of a package or to send to one or two individual testers, for example.
Note:
You can also archive a package automatically after installing the package. For more information, go to Install Inbound Releases Screen.
Note:
An alternative way of creating a backup is using the pub.packages:backupPackage service. However, when you use the backupPackage service, the package metadata of the backed up package is the same as the original package. For example, the creation timestamp will reflect the creation timestamp of the original package. For more information, see the webMethods Integration Server Built-In Services Reference.
To open the Archive Package Screen, go to the Packages Management Screen. In the row containing the package you want to archive, click the Archive icon
Click Return to Package Management to return to the main package management screen.
Specify Files for the Archive
Use this section to specify which files you want to include in the archive file.
Files available in packageName
Displays the files that make up the package. You may select specific files in this list to include or exclude from the archive.
Files to Include
Specifies which files are to be included in the archive. Select one of the following options.
Select...
If...
All Files
You want to archive all files shown in the file list.
Selected files
You want to archive only the selected files in the file list.
All except selected files
You want to archive all files except those selected in the file list.
Files specified by filter:
You want to archive only those files matching a particular pattern-matching string. You may use the * character to represent any sequence of characters (e.g., *.java would archive only the Java source files in the package).
All except files specified by filter:
Select this option if you want to archive all files except those matching a particular pattern-matching string. You may use the * character to represent any sequence of characters (e.g., *.java would archive everything except Java source files).
Files to Delete from Target Package
Use this section to identify files you want to delete from the target package. Separate each entry with a semicolon (;). When the server creates the archive file, it deletes the files you specified from the target package.
Package Release Information
Use this section to identify and describe the archive.
Archive Type
Specifies whether the archive is a complete or partial copy of the package. Select one of the following:
Select...
If...
Full
The archive contains a complete package. A full archive is one that contains all the files in the package. A full archive can be installed on a server that does not already have the package. (If you install it on a server that already has the package, it will completely replace the earlier version.)
Patch
The archive contains a partial package. A patch contains one or more files from a package, but does not represent a complete package. It is a way to save a specific set of files associated with a package. A patch release can only be installed on a server that has a complete copy of the package installed.
Archive Name
Specifies the name of the archive. The archive name is also the name of the zip file that the server generates and stores in the Integration Server_directory \instances\instance_name\replicate\outbound directory. Type a name that identifies this archive uniquely. Do not type an extension. The server appends the extension .zip to the name when it creates the archive file.
Brief Description
Optional. Specifies a brief comment describing the archive. You can use this field for any sort of descriptive information--for example, "Week 001215 Backup" or "Preview version for training group." The description will appear on the Package Description Screen when this archive is reinstalled.
Version
Specifies the version of the package that you are archiving. You must specify a version number in the form n.n.n (e.g., 2.0, 2.3, 2.3.1). Because the server assumes that version numbers increase over time, make sure every new version of a package has a version number that is greater than the previous one.
Build Number
Optional. Specifies a number that you can use to distinguish different generations of the package within the same version. Must be a numeric value that increases over time.
Previous Patches Included
Optional. Describes which patches this archive includes. A patch can be identified with any name or number that is meaningful to your site and/or the users of the package. Typically, a patch is identified using a code (e.g., TMgr-031501 or FIX-10002) or a number (e.g., 100049) that represents a particular fix or set of fixes.
Recommendations for Subscriber Install
Use this section to inform subscribers of this package's installation requirements.
webMethods Integration Server
Optional. Specifies the version of the webMethods Integration Server that must be used with this release.
Minimum Version of JVM
Specifies the minimum version of the Java Virtual Machine that must be used with this release.
Required Settings Creating a Patch
Specifies the version of the package to which a patch applies.
Version of Target Package
If this archive contains a patch (partial package), specify the version number of the package that it modifies. The server will allow the patch to be applied only to the version you specify.
