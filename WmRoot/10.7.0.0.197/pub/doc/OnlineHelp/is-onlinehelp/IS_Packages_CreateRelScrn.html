Create Release Screen
Screen Heading: Packages > Publishing > Create Release > packageName
Use the Create Release screen to create a new release (distribution file) for a package.
To open the Create Releases screen, go to the Publishing Packages Screen and click Create and Delete Releases. From the Create and Delete Releases Screen, click the Create Release link next to the package name.
Click Return to Create and Delete Releases to return to list of available releases.
Specify Files for the Release
Use this section to specify which files you want to include in the release.
Files available in packageName
Displays all the files that make up the package. You may select specific files in this list to include or exclude them from the release.
Files to Include
Specifies which files are to be included in the release. Select one of the following options.
Select...
If...
All Files
You want all files shown in the file list to be included in the release.
Selected files
You want only the files you select in the file list to be included in the release.
All except selected files
You want all files except those you select in the file list to be included in the release.
Files specified by filter:
You want to include in the release only those files matching a particular pattern-matching string. You may use the * character to represent any sequence of characters (e.g., *.java would archive only the java source files in the package).
All except files specified by filter:
Select this option if you want to include in the release all files except those matching a particular pattern-matching string. You may use the * character to represent any sequence of characters (e.g., *.java would archive everything except java source files).
Files to Delete from Target Package
Use this section to identify files you want to delete from the target package. Separate each entry with a semicolon (;). When the subscribing server installs the package, it deletes the files you specified from the target package.
Package Release Information
Use this section to identify and describe the release.
Release Type
Specifies whether this is a completely new release of the package or is a partial upgrade (patch) to an existing release. Select one of the following:
Select...
If...
Full
The release represents a complete package. A full release is one that contains all the files that make up a package. A full release can be installed on a server that does not already have that package.
Patch
The release represents a partial package. A patch contains one or more files from a package, but does not represent a complete release. A patch is a way to distribute a specific set of files associated with a package. A patch can only be installed on a server that already has a complete copy of the package.
Release Name
Specifies the name of the release. The release name is also the name of the zip file that the server generates and stores in the Integration Server_directory \instances\instance_name\replicate\outbound directory. Type a name that identifies this release uniquely. Do not type an extension. The server appends the extension .zip to the name when it creates the distribution file.
Brief Description
Optional. Specifies a brief comment describing the release. You can use this field for any sort of descriptive information. For example, "Orders 2001" or "Correction to prevent duplicate GL entries". The description will appear on the Package Description Screen when the release is installed.
Version
Specifies the version of the release that you are creating. You must assign a version number in the form of n.m (e.g., 2.0, 2.3, 2.3.1). Because the server assumes that version numbers increase over time, make sure you give every new version of a package a number that is greater than the previous one.
Build Number
Optional. Specifies a number that you can use to distinguish different generations of the package within the same version. Must be a numeric value that increases with each release within a version.
Previous Patches Included
Optional. Describes which patches this release includes. A patch can be identified using any name or number that is meaningful to your site and/or the users of the package. Typically, a patch is represented by a code (example: TMgr-031501 or FIX-10002) or a number (e.g., 100049) that represents a particular fix or set of fixes.
Recommendations for Subscriber Install
Use this section to inform subscribers of this package's installation requirements.
webMethods Integration Server
Optional. Specifies the version of webMethods Integration Server that must be used with this release.
Minimum Version of JVM
Specifies the minimum version of the Java Virtual Machine that must be used with this release. When the administrator installs the package, the server checks the version of the JVM it is running. If it is running a lower major version of the JVM, the server installs the package but does not activate it.
Required Settings for Creating a Patch
Specifies the version of the package to which a patch applies.
Version of Target Package
If this release is a patch (partial package), specify the version of the package that it modifies. The server will allow the patch to be applied only to the version of the package you specify.
